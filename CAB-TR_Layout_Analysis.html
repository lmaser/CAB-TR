2. Diagnóstico — Causas raíz del solapamiento

2.1 ROOT CAUSE getValueAreaFor() calcula contra getWidth(), no contra el límite de columna

Este es el problema principal. El código actual:

juce::Rectangle<int> getValueAreaFor (const juce::Rectangle<int>& barBounds) const
{
    constexpr int valuePadPx    = 8;        // fijo
    constexpr int valueWidthPx  = 200;      // fijo — ¡puede ser más ancho que la columna!
    constexpr int valueHeightPx = 24;
    constexpr int rightMarginPx = 10;       // fijo

    const int valueX = barBounds.getRight() + valuePadPx;
    const int maxW = jmax(0, getWidth() - valueX - rightMarginPx);
    //                        ^^^^^^^^^^
    //  Se usa el ancho TOTAL del editor, no el de la columna.
    //  La columna A tiene derecho real a: colWidth - margin*2
    //  Pero el texto puede extenderse hasta getWidth() - 10
    const int valueW = jmin(valueWidthPx, maxW);
    ...
}

Consecuencia directa:

  • Columna A: El texto puede extenderse visualmente sobre la columna B, porque getWidth() - valueX - 10 da un ancho enorme.
  • Columna B: El texto puede extenderse sobre la columna C.
  • Columna C: Menos afectada (sólo el margen derecho final cuenta), pero aún puede recortarse mal.
Ejemplo: editor width = 900px → colWidth = 300px Columna A (x: 0-300) Columna B (x: 300-600) Columna C (x: 600-900) ┌──────────┬─────────────────┬──────────┬─────────────────┬──────────┬──────────┐ │ margin │ area │ margin │ area │ margin │ area │ │ 10 │ 280 │ 10 │ 280 │ 10 │ 280+ │ ├──────────┴─────────────────┼──────────┴─────────────────┼──────────┴──────────┤ │ │ │ │ │ sliderW = 280 * 0.55 │ sliderW = 280 * 0.55 │ │ │ = 154px │ = 154px │ │ │ │ │ │ │ valueX = 10 + 154 + 8 │ valueX = 310 + 154 + 8 │ │ │ = 172 │ = 472 │ │ │ │ │ │ │ maxW = 900 - 172 - 10 = 718 │ maxW = 900 - 472 - 10 = 418 │ │ │ capado a 200 → OK en │ capado a 200 → OK en │ │ │ tamaño grande, PERO: │ tamaño grande, PERO: │ │ │ │ │ │ │ valueX + 200 = 372 │ valueX + 200 = 672 │ │ │ → INVADE columna B (300+) │ → INVADE columna C (600+) │ │ │ │ │ │ └────────────────────────────┴────────────────────────────┴─────────────────────┘

Resultado visual: Incluso a 900px de ancho, los textos de la columna A como "10000 ms END" (≈120px a 32pt bold) se extienden más allá del límite de la columna A y solapan con los sliders de la columna B.

2.2 BUG El padding 8px y el cap de 200px son fijos, no proporcionan al ancho de columna

A 800px (ancho mínimo), cada columna mide ~266px. Con margin 10 → area de 246px. Slider = 246 * 0.55 = 135px. Espacio restante para texto: 246 - 135 - 8 = 103px.

Un texto como "10000 ms START" a 32pt bold mide ≈180px. El cascade baja a "10000 ms" (≈100px) y luego a "10000" (≈55px), pero el ancho de la fuente a 24pt (75% floor) puede ser ajustado. El sistema de cascade funciona, pero los textos se están recortando en tamaños intermedios donde caben en maxW global pero no en la columna.

2.3 BUG tryDrawLegend usa maxW calculado contra getWidth(), no contra columnRight

El problema del punto 2.1 tiene un efecto dominó: tryDrawLegend recibe un area lo suficientemente ancha como para que el texto "completo" quepa → no activa el cascade a short/intOnly. El texto se pinta completo y cruza el borde de la columna.

Si el area se limitara a los bounds de la columna, el cascade se activaría correctamente y mostraría el texto corto o solo el número cuando no hay espacio.

2.4 MINOR Slider width 55% está ajustado para 1 columna, no para 3

En los otros TR plugins (DISP-TR, ECHO-TR, FREQ-TR), que son de columna única, se usa barW = editorW * 0.455. En CAB-TR, con 3 columnas, se usa area.getWidth() * 0.55. Esto deja un 45% de la columna para texto, que con margin y padding se reduce significativamente.

4. Propuestas de solución

Propuesta A — Limitar valueArea al bounds de la columna RECOMENDADA

Esfuerzo: bajo Riesgo: bajo Impacto: resuelve 100% del overlap

Descripción

Modificar getValueAreaFor() para que el límite derecho del texto sea el borde derecho de la columna en lugar de getWidth(). Esto requiere que cada slider "sepa" a qué columna pertenece, o pasar el columnRight como parámetro.

Cambios concretos

// ANTES:
juce::Rectangle<int> getValueAreaFor (const juce::Rectangle<int>& barBounds) const
{
    const int valueX = barBounds.getRight() + 8;
    const int maxW = jmax(0, getWidth() - valueX - 10);
    const int valueW = jmin(200, maxW);
    ...
}

// DESPUÉS:
juce::Rectangle<int> getValueAreaFor (const juce::Rectangle<int>& barBounds,
                                       int columnRight) const
{
    constexpr int valuePadPx    = 8;
    constexpr int valueHeightPx = 24;
    constexpr int rightMarginPx = 6;

    const int valueX = barBounds.getRight() + valuePadPx;
    const int maxW   = jmax(0, columnRight - valueX - rightMarginPx);
    const int y      = barBounds.getCentreY() - (valueHeightPx / 2);

    return { valueX, y, maxW, valueHeightPx };
}

Los call sites (paint + hit-test) necesitan pasar el columnRight correcto para cada loader. Se puede calcular en resized() y cachear en un array int columnRight_[3].

Efecto en el cascade

Con el valueArea correctamente limitado a la columna, el sistema tryDrawLegend ya existente activará automáticamente el cascade (full → short → intOnly) cuando el texto no quepa. No se necesita cambiar la lógica de cascade.

Con propuesta A implementada (900px): ┌───────────────────┬───────────────────┬────────────────────┐ │ col A (0-300) │ col B (300-600) │ col C (600-900) │ │ │ │ │ │ [═══════] 10000ms │ [═══════] 10000ms │ [═══════] 10000 ms │ │ ^^^^^^^^^│ ^^^^^^^^^│ ^^^^^^^^^│ │ (short cascade) │ (short cascade) │ (fits full → col)│ │ limitado a 300 │ limitado a 600 │ limitado a 900 │ └───────────────────┴───────────────────┴────────────────────┘ ¡Sin solapamiento!

Ventajas

  • Solución mínimamente invasiva — cambia 1 función + pasar 1 parámetro extra
  • El cascade full → short → intOnly ya funciona bien, solo necesita el área correcta
  • Compatible con todos los tamaños (800-2000)
  • Mantiene la apariencia actual; solo se activa el cascade cuando realmente no cabe

Contras

  • A 800px mínimo, la mayoría de textos aparecerán como "short" o "intOnly" (pero se leen)

Propuesta B — Propuesta A + valuePad y ratio proporcionales RECOMENDADA

Esfuerzo: medio Riesgo: bajo Impacto: resuelve overlap + mejora scaling

Descripción

Implementar Propuesta A más adoptar el patrón proporcional de los otros TR plugins. El slider ratio y padding se calculan en función del ancho de columna (actuando como "editorW" en los otros plugins).

Cambios concretos

juce::Rectangle<int> getValueAreaFor (const juce::Rectangle<int>& barBounds,
                                       int columnRight, int valuePad) const
{
    const int valueX = barBounds.getRight() + valuePad;
    const int maxW   = jmax(0, columnRight - valueX - 6);
    ...
}

Propuesta C — Propuesta A/B + sistema rhythm vertical (como DISP-TR)

Esfuerzo: medio-alto Riesgo: medio

Descripción

Además del fix horizontal, adoptar el sistema rhythm de DISP-TR para la distribución vertical. Esto proporciona una experiencia más suave al redimensionar verticalmente.

// Vertical rhythm como DISP-TR pero por columna
const int rhythm    = jlimit(6, 16, (int) std::round (area.getHeight() * 0.018));
const int nomBarH   = jlimit(14, 120, rhythm * 6);
const int nomGapY   = jmax(4, rhythm * 4);

// Degrado gradual: primero reduce gaps, luego altura de barra
while (stackHeight() > available && gapY > 4)   --gapY;
while (stackHeight() > available && barH > 14)   --barH;

Ventajas

  • Transiciones más suaves al redimensionar verticalmente
  • Consistencia visual con la familia TR
  • Los gaps se comprimen antes que los bars → los sliders mantienen usabilidad

Contras

  • Más código a modificar (layoutIRSection completa)
  • Requiere testing visual extensivo en ambos modos (collapsed/expanded)
  • No es estrictamente necesario para resolver el solapamiento horizontal

Propuesta D — Aumentar ancho mínimo a 1080px

Esfuerzo: mínimo Riesgo: limitaciones UX

Descripción

Cambiar setResizeLimits(800, ...) a setResizeLimits(1080, ...) para que cada columna tenga mínimo ~360px (como el ancho mínimo de los otros TR plugins single-column).

setResizeLimits (1080, 540, 2000, 1200);

Ventajas

  • Un solo cambio de línea
  • A 1080px las columnas tienen 360px, equivalente al min de DISP-TR single-column
  • Reduce la gravedad del solapamiento en los tamaños más pequeños

Contras

  • No resuelve el problema de raíz — el texto aún puede cruzar columnas
  • Usuarios con pantallas pequeñas no podrán reducir el plugin
  • Debe combinarse con Propuesta A o B para ser completa

Propuesta E — Modo "compact" con texto sobre el slider (stacked layout)

Esfuerzo: alto Riesgo: alto

Descripción

Cuando el ancho de columna baja de un umbral (ej. 300px), cambiar a un layout "stacked" donde el texto del valor se pinta encima del slider bar en lugar de a su derecha. Esto elimina la competencia horizontal por espacio.

Ventajas

  • Elimina completamente el problema de solapamiento horizontal
  • Los sliders pueden usar el 100% del ancho de columna
  • Legibilidad total del texto incluso a tamaños mínimos

Contras

  • Requiere reestructuración significativa de layoutIRSection y paint
  • Reduce el espacio vertical (cada slider necesita ~2x de altura)
  • Rompe la consistencia visual con los otros TR plugins
  • Con 8 sliders en collapsed, podría no caber verticalmente a 500px de alto

6. Recomendación

🏆 Propuesta B (A + proporcional)

Es la mejor relación esfuerzo/resultado:

  1. Paso 1: Aplicar Propuesta A — limitar getValueAreaFor() al borde de columna. Esto resuelve el solapamiento inmediatamente.
  2. Paso 2: Ajustar slider ratio (55→50%) y hacer valuePad proporcional. Mejora el balance visual.
  3. Opcional: Combinar con Propuesta D (subir min width a 1000-1080px) como safety net.

El sistema de cascade full → short → intOnly ya implementado se encargará automáticamente de la adaptación textual una vez que las áreas estén correctamente limitadas.

7. Resumen ejecutivo

Problema raíz

getValueAreaFor() calcula el espacio disponible contra getWidth() (ancho total del editor) en lugar del borde derecho de la columna respectiva. Esto permite que el texto de cada columna invada la columna siguiente.

Fix mínimo viable

Pasar columnRight a getValueAreaFor() y usarlo como límite. El cascade existente hará el resto. ~30 líneas de cambio.

Fix recomendado

Propuesta B: fix de columnRight + ajuste proporcional del ratio slider/pad. ~50 líneas. Consistente con la familia TR.
