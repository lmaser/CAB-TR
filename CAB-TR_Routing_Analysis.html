§2 — Cadena de procesamiento por loader (processOne)

Cada loader ejecuta la misma cadena de 10 pasos, independientemente del routing:

processOne — Cadena interna de cada loader
DRY save → Mode IN → CONVOLUTION → HP filter → LP filter → DISTANCE
→ PAN → DELAY → ANGLE (Fred) → CHAOS → OUT GAIN → Mode OUT → MIX dry/wet
Orden crítico
PAN se aplica DENTRO de processLoader (paso 5), y Mode OUT se aplica DESPUÉS de processLoader. Esto significa que Mode OUT opera sobre una señal ya paneada en el campo estéreo.

§3 — Mode In / Mode Out — Opciones actuales

ValorLabelFórmulaEfecto
0 L+R sin cambio Pass-through estéreo (default)
1 MID L = R = (L+R) × 0.707 Extrae componente centro → ambos canales iguales (mono)
2 SIDE L = R = (L−R) × 0.707 Extrae componente ancho → ambos canales iguales (mono)
⚠ Limitación clave
Tanto MID como SIDE colapsan a mono: ponen la misma señal en L y R. Después de aplicar Mode Out, se pierde toda información espacial del PAN. Además, solo hay 3 opciones — no hay LEFT, RIGHT, ni M/S encode sin colapso.

§5 — Análisis matemático

5.1 — Señal actual tras PAN y suma (Route 1, todo Mode Out = L+R)

Después de convolución con IR mono, PAN aplica constant-power panning:

PAN constant-power:
L_gain = cos(pan × π/2)   R_gain = sin(pan × π/2)

Loader A (pan=0.0): LA = 1.000 × SA , RA = 0.000 × SA
Loader B (pan=0.5): LB = 0.707 × SB , RB = 0.707 × SB
Loader C (pan=1.0): LC = 0.000 × SC , RC = 1.000 × SC

Suma paralela (con comp. −4.77 dB):
L_sum = (SA + 0.707 × SB) × 0.577
R_sum = (0.707 × SB + SC) × 0.577

5.2 — Descomposición M/S de la suma

Mid = (L + R) / √2 :
Mid = (SA + 1.414 × SB + SC) × 0.408
↑ Contiene A, B y C — B NO está aislado en mid

Side = (L − R) / √2 :
Side = (SA − SC) × 0.408
↑ Solo A y C — B desaparece del side completamente ✓
Hallazgo clave
Cuando B está perfectamente centrado (pan=0.5), su contribución al Side es exactamente cero, porque LB = RB → (LB − RB) = 0.

Pero en el Mid, A y C sí se filtran: sus señales hard-panned contribuyen como (SA + SC) × 0.408. Para aislar B perfectamente en mid, necesitamos que A y C no pasen por el bus mid.

5.3 — ¿Y si usamos Mode Out per-loader?

Config.Mode Out AMode Out BMode Out CResultado
Intento 1 SIDE MID SIDE Todos colapsan a mono → pierde toda la imagen estéreo
Intento 2 L+R MID L+R B colapsa a mono, A+C mantienen PAN → B aún se mezcla con A+C en mid de la suma
Intento 3 SIDE L+R SIDE A: (L−0)/√2 = A/√2 mono. C: (0−R)/√2 = −C/√2 mono. Imagen destruida.
✗ Conclusión
Con las 3 opciones actuales de Mode Out (L+R, MID, SIDE) no hay combinación que permita aislar loaders en buses M/S separados cuando se trabaja en paralelo. El problema raíz: MID y SIDE colapsan a mono, destruyendo la separación espacial que PAN había creado.

§6 — Propuestas de solución

Propuesta A — Global Post-Sum Output Mode

Fácil

Un nuevo parámetro global que se aplica DESPUÉS del routing/suma y ANTES de Match Tilt EQ:

ValorLabelEfecto post-suma
0STSin cambio (comportamiento actual)
1MIDL = R = (L+R) × 0.707 — solo componente centro
2SIDEL = R = (L−R) × 0.707 — solo componente ancho
Flujo con Propuesta A
INPUT → ROUTING (A|B|C) → Σ parallel sum → OUTPUT MODE ★ → Tilt EQ → OUTPUT

✓ Ventajas

  • Extremadamente simple: ~10 líneas de DSP
  • 1 solo parámetro nuevo
  • SIDE post-suma aísla B perfectamente (B centrado → 0 en side)
  • Útil para monitoreo rápido de imagen estéreo

✗ Limitaciones

  • MID post-suma no aísla B (A y C se filtran en mid)
  • No permite routing por bus M/S per-loader
  • Es todo-o-nada: o escuchas mid, o side, no combinación selectiva
Impacto en código:
~10 líneas DSP · 1 parámetro · 1 combo UI

Propuesta B — Mode Out extendido: LEFT / RIGHT

Fácil

Añadir 2 nuevas opciones a Mode Out per-loader:

ValorLabelEfectoEstado
0L+RSin cambioExistente
1MIDL = R = (L+R) × 0.707Existente
2SIDEL = R = (L−R) × 0.707Existente
3→ LR = 0, L sin cambio★ Nuevo
4→ RL = 0, R sin cambio★ Nuevo

✓ Ventajas

  • Simple: ~8 líneas en applyMode
  • Útil con PAN parcial (no solo hard-pan)
  • No requiere nuevos parámetros — solo extiende rango existente

✗ Limitaciones

  • Redundante con PAN hard-left / hard-right
  • No resuelve separación M/S por bus
  • Valor limitado como herramienta creativa
Impacto en código:
~8 líneas DSP · 0 parámetros nuevos · solo cambio de rango en combos

Propuesta C — Per-Loader "Sum Bus" (M/S Bus Routing)

Media ★ RECOMENDADA

Nuevo parámetro per-loader: SUM BUS — controla cómo la salida de ese loader se inyecta en la suma paralela. Solo tiene efecto en modos con ramas paralelas (Routes 1, 2, 3).

ValorLabelComportamiento en la suma
0 ST Suma directa L+R (actual) — contribuye a todo el campo estéreo
1 → M Se colapsa a mono: mono = (L+R)/2 → se suma a ambos canales iguales (solo mid)
2 → S Se extrae diferencia: side = (L−R)/2 → se suma como L += side, R −= side (solo side)

Demostración matemática — el escenario del usuario

Config: A(pan=0,bus=→S), B(pan=0.5,bus=→M), C(pan=1,bus=→S)

Tras processOne + PAN:
A: LA = SA, RA = 0
B: LB = 0.707 × SB, RB = 0.707 × SB
C: LC = 0, RC = SC

Bus routing:
A → SIDE bus: sideA = (SA − 0) / 2 = SA/2
B → MID bus: monoB = (0.707SB + 0.707SB) / 2 = 0.707 × SB
C → SIDE bus: sideC = (0 − SC) / 2 = −SC/2

Acumuladores:
mid_total = 0.707 × SB
side_total = SA/2 + (−SC/2) = (SA − SC) / 2

Reconstrucción a estéreo:
L = mid_total + side_total = 0.707 × SB + (SA − SC) / 2
R = mid_total − side_total = 0.707 × SB − (SA − SC) / 2

Verificación M/S del output:
Mid(output) = (L+R)/√2 = √2 × 0.707 × SB = SB → ✓ ¡SOLO B!
Side(output) = (L−R)/√2 = √2 × (SA−SC)/2 = (SA−SC)/√2 → ✓ ¡SOLO A y C!
✓ Separación perfecta
Con Sum Bus → M en B y → S en A+C, la salida tiene:
  • Mid: Exclusivamente SB — sin contaminación de A ni C
  • Side: Exclusivamente (SA − SC) — sin contaminación de B
El resultado es un campo estéreo donde B está perfectamente centrado y A/C crean la anchura, sin mez entre buses. Sí un DAW downstream aplica M/S decode, obtiene extraccion perfecta.

¿Cómo afecta a cada routing mode?

RouteTopología¿Sum Bus aplica?¿Dónde?
0 A→B→C No (1 solo camino) En serie no hay suma — Sum Bus no tiene efecto
1 A|B|C Sí — 3 loaders Cada loader puede asignarse a ST, →M o →S antes de la suma final
2 A→B | C Sí — 2 ramas La rama serie (A→B) y C pueden asignarse a buses distintos
3 A | B→C Sí — 2 ramas A y la rama serie (B→C) pueden asignarse a buses distintos
Nota sobre Routes 2/3
En Routes 2 y 3, el Sum Bus aplica a cada rama paralela (no a cada loader individual dentro de la serie). El último loader de la cadena serie determina el bus. En Route 2: bus del path (A→B) se toma de B, y bus de C es independiente. En Route 3: bus de A es independiente, y bus del path (B→C) se toma de C.

Alternativa: Ignorar Sum Bus en Routes 0/2/3 y reservarlo exclusivamente para Route 1. Esto simplifica la implementación y el concepto es más claro: "M/S bus routing = solo en modo full parallel". Si lo necesitas en Routes 2/3, se implementa después.

✓ Ventajas

  • Separación M/S perfecta per-loader
  • Compatible con todos los parámetros existentes (PAN, Mode In/Out, etc.)
  • Concepto intuitivo: "este cab va al centro / a los lados"
  • Único en el mercado — ningún otro cab sim ofrece M/S bus routing

✗ Consideraciones

  • 3 nuevos parámetros (1 por loader) — requieren automación/estado
  • La compensación de ganancia varía según la combinación de buses
  • Sin efecto en Route 0 (serie) — el UI podría ocultar/deshabilitar
  • La interacción con Mode Out existente requiere documentación
Impacto en código:
~35 líneas DSP (mayormente en suma paralela) · 3 parámetros nuevos · 3 combos UI

§9 — Recomendación final

Implementación recomendada: Propuesta C sola
  1. Implementar Sum Bus per-loader (Propuesta C) solo para Route 1 (A|B|C) inicialmente. Es la configuración más usada y donde la funcionalidad tiene más sentido.
  2. 3 nuevos parámetros: sum_bus_a, sum_bus_b, sum_bus_c — cada uno con opciones ST / → M / → S
  3. ~35 líneas de DSP en la sección de suma paralela de processBlock. El resto del código (processOne, processLoader, Mode In/Out) no cambia.
  4. Extensión futura: Si se necesita, extender a Routes 2/3 (ramas paralelas) y añadir la Propuesta A (global output mode) como herramienta de monitoreo.

Pseudocódigo de la suma M/S-aware (Route 1)

// — Dentro del bloque if (route == 1) — A|B|C paralelo —

// 1. Procesar cada loader como siempre
if (enableA) processOne (stateA, tempBufferA, 0, modeInA, modeOutA, mixA);
if (enableB) processOne (stateB, tempBufferB, 1, modeInB, modeOutB, mixB);
if (enableC) processOne (stateC, tempBufferC, 2, modeInC, modeOutC, mixC);

// 2. Acumular según Sum Bus
buffer.clear();
float midAccum = 0, sideAccum = 0;  // per-sample, dentro del loop

for (int i = 0; i < numSamples; ++i)
{
    float sumL = 0, sumR = 0;
    float midBus = 0, sideBus = 0;
    bool hasMSBus = false;

    // Para cada loader activo:
    auto accumulate = [&] (auto& buf, int bus) {
        float l = buf.getReadPointer(0)[i];
        float r = buf.getReadPointer(1)[i];

        if (bus == 0)      // ST — suma directa
        {   sumL += l; sumR += r;   }
        else if (bus == 1) // → M — solo contribuye a mid
        {   midBus += (l + r) * 0.5f; hasMSBus = true;   }
        else               // → S — solo contribuye a side
        {   sideBus += (l - r) * 0.5f; hasMSBus = true;   }
    };

    if (enableA) accumulate (tempBufferA, sumBusA);
    if (enableB) accumulate (tempBufferB, sumBusB);
    if (enableC) accumulate (tempBufferC, sumBusC);

    // 3. Reconstruir estéreo: L = mid + side, R = mid − side
    if (hasMSBus)
    {   sumL += midBus + sideBus;  sumR += midBus - sideBus;   }

    buffer.getWritePointer(0)[i] = sumL;
    buffer.getWritePointer(1)[i] = sumR;
}

// 4. Compensación
buffer.applyGain (1.0f / std::sqrt ((float) numActive));
¿Tiene sentido?
Absolutamente sí. El escenario que describes (3 cabs mono → L/C/R → control M/S independiente) es un caso de uso real en producción de guitarra. La técnica de colocar cabs distintos en los lados y uno neutral en el centro es común, y poder controlar cuánto "centro" vs "anchura" aporta cada cab es una herramienta poderosa que actualmente no ofrece ningún cab sim del mercado que yo conozca.

El coste de implementación es moderado (~35 líneas DSP, 3 params, 3 combos) y no rompe ningún comportamiento existente (Sum Bus = ST por defecto = comportamiento actual).
